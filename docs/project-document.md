---

## 1. 프로젝트 개요

### 한 줄 소개
> "한국어로 질문하면 SQL 쿼리로 바꿔주는 AI 번역기"

### 배경 및 목적

데이터를 조회하거나 분석하려면 SQL이라는 전문 언어를 알아야 합니다.
하지만 기획자, 마케터, 운영팀처럼 데이터가 필요하지만 SQL을 모르는 구성원이 많습니다.

이 도구는 그런 분들이 **"지난달 가입한 VIP 회원 보여줘"** 처럼 자연스러운 말로 입력하면
AI가 알아서 SQL로 변환해주는 서비스입니다.

### 주요 사용 대상

| 대상 | 활용 방법 |
|------|-----------|
| SQL을 모르는 기획자/마케터 | 자연어로 원하는 데이터 조회 쿼리 생성 |
| 주니어 개발자 | 복잡한 쿼리 작성 시 초안 참고 |
| DBA / 시니어 개발자 | 빠른 쿼리 초안 작성 후 수정 |

## 2. 핵심 기능

### 2-1. 자연어 → SQL 변환

한국어 문장을 입력하면 AI가 표준 SQL로 변환합니다.
변환 결과로 SQL 쿼리, 한글 설명, 사용된 테이블명을 함께 제공합니다.

```
입력: "지난 30일간 청구 금액이 100만원 이상인 목록을 최근 날짜 순으로 보여줘"

출력 SQL:
SELECT * FROM PLUSCMS.TA_BILLING
WHERE BILL_DT >= CURRENT_DATE - INTERVAL '30 days'
  AND BILL_AMT >= 1000000
ORDER BY BILL_DT DESC;

설명: 최근 30일 내 청구 금액이 100만원 이상인 건을 최근 날짜 기준으로 정렬
```

### 2-2. 테이블 컨텍스트 (스키마 등록)

실제 사용 중인 DB 테이블 구조를 미리 등록해두면
AI가 추측 없이 **실제 테이블명과 컬럼명**으로 정확한 SQL을 생성합니다.

- Java 엔티티 코드를 그대로 붙여넣기 가능
- 직접 스키마 형식으로 입력도 가능
- 브라우저에 저장되어 다음 접속 시에도 유지

### 2-3. SQL 복사 버튼

생성된 SQL을 클릭 한 번으로 클립보드에 복사합니다.
복사 후 DBeaver, SQL Developer 등 DB 툴에 바로 붙여넣을 수 있습니다.

### 2-4. 분할 화면 레이아웃

입력창과 결과창이 좌우로 나뉘어 있어 변환 전후를 한 화면에서 확인할 수 있습니다.

## 4. 기술 구성 (비개발자용 설명)

### 전체 흐름

```
사용자 입력
    ↓
브라우저 (Next.js 앱)
    ↓  API 요청 (HTTPS)
서버 (Next.js API Route)
    ↓  API 요청 (Groq 서버로)
Groq AI 서버 (llama-3.3-70b 모델)
    ↓  JSON 응답
서버 → 브라우저 → 화면 표시
```

### 각 구성요소 역할

| 구성요소 | 쉬운 설명 |
|----------|-----------|
| **Next.js** | 웹 앱을 만드는 프레임워크. 화면(프론트엔드)과 서버(백엔드) 역할을 동시에 담당 |
| **React** | 화면의 버튼, 입력창 등 UI 요소를 만드는 라이브러리 |
| **Tailwind CSS** | 화면 디자인(색상, 여백, 크기 등)을 빠르게 적용하는 도구 |
| **Groq API** | AI 모델을 호출할 수 있는 서비스. 무료 tier 제공 |
| **llama-3.3-70b** | Groq에서 제공하는 AI 모델. 자연어 이해 및 SQL 생성 담당 |
| **Vercel** | 웹 앱을 인터넷에 배포하는 플랫폼. GitHub 연동으로 자동 배포 |

### API 키 보안 구조

```
❌ 잘못된 방식:
   브라우저 → Groq API (API 키가 소스코드에 노출)

✅ 현재 구조:
   브라우저 → 우리 서버(API Route) → Groq API
                    ↑
              API 키는 서버에만 보관
              외부에서 절대 보이지 않음
```

사용자는 API 키를 볼 수 없고, 로그인도 필요 없습니다.

## 6. 설계 결정 이유

### Q. 왜 Vercel인가요? GitHub Pages는 안 되나요?

이 앱은 API 키를 서버에서 안전하게 관리해야 합니다.
GitHub Pages는 서버 기능이 없어 API 키가 소스코드에 노출되는 보안 문제가 생깁니다.
Vercel은 서버리스 함수를 지원하고 무료로 사용 가능해서 선택했습니다.

### Q. 왜 Groq인가요?

| 서비스 | 비용 | 속도 | 비고 |
|--------|------|------|------|
| OpenAI GPT-4 | 유료 | 보통 | API 키 필수 |
| Anthropic Claude | 유료 | 보통 | API 키 필수 |
| **Groq** | **무료 tier** | **매우 빠름** | llama 모델 제공 |

무료로 사용 가능하고 응답 속도가 빠른 점에서 Groq을 선택했습니다.

### Q. 테이블 정보는 어디에 저장되나요?

사용자 브라우저의 `localStorage`에 저장됩니다.
서버나 DB에 저장되지 않아 개인 정보 보호 측면에서 안전합니다.
단, 다른 기기나 다른 브라우저에서는 공유되지 않습니다.

## 7. 향후 방향성

> 현재 이 도구는 "SQL을 만들어주는 번역기" 수준입니다.
> 최종 목표는 **"비개발자가 자연어로 업무 데이터를 직접 추출하는 사내 데이터 도구"** 입니다.

---

### 7-1단계 — AI 정확도 개선 (즉시 적용 가능)

현재 자연어 해석 품질을 높이기 위한 방법입니다.

| 방법 | 설명 | 효과 |
|------|------|------|
| **프롬프트 규칙 추가** | 날짜 처리, NULL 처리, 정렬 기본값 등 SQL 규칙 명시 | 높음 |
| **예시 추가 (Few-shot)** | 좋은 질문/답변 쌍을 AI에게 미리 보여줌 | 높음 |
| **DB 방언 명시** | Oracle / MySQL 등 사용 DB를 프롬프트에 고정 | 중간 |
| **모델 교체** | llama → Claude 3.5 Sonnet 등 고성능 모델 사용 | 높음 (유료) |

---

### 7-2단계 — 공용 스키마 내장 (팀 공유 기반)

지금은 사용자가 각자 브라우저에 스키마를 저장합니다.
→ 관리자가 회사 주요 테이블을 코드에 미리 등록해두면 팀원 누구나 즉시 사용 가능

```
현재: 개인 브라우저 저장 (기기마다 다름)
개선: 앱에 공용 테이블 내장 → 전 팀원 동일한 스키마 사용
```

**필요한 작업**: 주요 테이블 스키마 정리 및 코드 등록

---

### 7-3단계 — DB 직접 연결 (핵심 전환점)

현재는 SQL을 복사해서 DBeaver 등에 직접 붙여넣어야 합니다.
→ 앱에서 실제 DB에 연결하여 결과를 바로 화면에 표시

```
현재: 자연어 → SQL 생성 → 사용자가 직접 실행
목표: 자연어 → SQL 생성 → 자동 실행 → 결과 테이블 표시 → CSV 다운로드
```

**고려사항**
- 회사 DB 방화벽/VPN 접근 허용 여부
- 읽기 전용 계정 사용 (SELECT만 허용, 데이터 수정 불가)
- 사내 서버 배포 (Vercel 대신 내부망 전용)

---

### 7-4단계 — VOC / 업무 데이터 특화

DB 연결이 완료되면 업무별 특화 기능을 추가합니다.

| 기능 | 내용 |
|------|------|
| 자주 쓰는 질문 템플릿 | "이번 달 VOC 현황", "미처리 건수" 등 버튼으로 제공 |
| 결과 테이블 표시 | 쿼리 실행 결과를 화면에 직접 표시 |
| CSV / 엑셀 다운로드 | 결과를 파일로 저장 |
| SELECT 전용 제한 | DELETE, DROP 등 위험 쿼리 생성 차단 |
| 역방향 변환 | SQL → 한글 설명 (다른 사람이 짠 쿼리 해석) |

---

### 7-5단계 — 권한 및 협업 도구

| 기능 | 내용 |
|------|------|
| 사내 계정 로그인 | SSO 연동으로 인증 |
| 부서별 테이블 접근 제한 | 인사팀은 인사 테이블만, 마케팀은 마케팅 테이블만 |
| 쿼리 공유 링크 | 생성된 쿼리를 URL로 공유 → 슬랙에 붙여넣기 |
| 즐겨찾기 | 자주 쓰는 쿼리 저장 및 재사용 |
| 사용 로그 | 누가 어떤 쿼리를 생성했는지 기록 (감사 목적) |

---

### 전체 로드맵 요약

```
현재        1단계        2단계        3단계        4~5단계
  │            │            │            │              │
SQL 생성  →  AI 정확도  →  공용 스키마 →  DB 연결   →  사내 데이터
  (복사 필요)   향상          내장          결과 표시      플랫폼
```

### 현실적인 시작점

> **바로 다음 한 걸음: 공용 스키마 파일 정리**
>
> 주요 테이블(VOC, 청구, 회원 등)의 스키마를 정리해서 코드에 내장하면
> 팀원 누구나 설정 없이 정확한 SQL을 바로 쓸 수 있습니다.
