## 1. 프로젝트 개요

### 한 줄 소개
> "우리 회사 DB 구조를 알고 있는 SQL 생성기"

### 배경 및 목적

데이터를 조회하거나 분석하려면 SQL이라는 전문 언어를 알아야 합니다.
하지만 기획자, 마케터, 운영팀처럼 데이터가 필요하지만 SQL을 모르는 구성원이 많습니다.

이 도구는 그런 분들이 **"지난달 가입한 VIP 회원 보여줘"** 처럼 자연스러운 말로 입력하면
AI가 **실제 우리 DB 테이블명과 컬럼명**으로 SQL을 바로 생성해주는 서비스입니다.

### 핵심 가치

단순한 자연어 → SQL 변환은 ChatGPT에게 물어봐도 됩니다.
이 도구의 차별점은 하나입니다.

> **우리 회사 테이블과 컬럼을 등록해두면, AI가 실제 컬럼명으로 바로 쓸 수 있는 SQL을 생성한다.**

DBA에게 "이 SQL 맞아요?" 물어볼 필요 없이, 생성된 쿼리를 바로 복사해서 사용할 수 있습니다.

### 주요 사용 대상

| 대상 | 활용 방법 |
|------|-----------|
| SQL을 모르는 기획자/마케터 | 자연어로 입력 → 실제 테이블 기반 쿼리 즉시 생성 |
| 주니어 개발자 | 복잡한 쿼리 초안 빠르게 작성 후 수정 |
| DBA / 시니어 개발자 | 반복적인 기본 쿼리 빠르게 생성 |

## 2. 핵심 기능

### 2-1. 자연어 → SQL 변환 (테이블 등록 시)

사용 중인 DB 테이블과 컬럼을 등록하고 선택하면, AI가 실제 테이블명과 컬럼명으로 SQL을 생성합니다.

```
입력: "지난 30일간 청구 금액이 100만원 이상인 목록을 최근 날짜 순으로 보여줘"
선택된 테이블: 청구 (PLUSCMS.TA_BILLING)

출력 SQL:
SELECT * FROM PLUSCMS.TA_BILLING
WHERE BILL_DT >= CURRENT_DATE - INTERVAL '30 days'
  AND BILL_AMT >= 1000000
ORDER BY BILL_DT DESC;

설명: 최근 30일 내 청구 금액이 100만원 이상인 건을 최근 날짜 기준으로 정렬
```

### 2-2. 테이블 미등록 시 변환 (플레이스홀더 모드)

테이블을 등록하지 않고 변환하면, AI가 `<테이블명>`, `<컬럼명>` 형태의 플레이스홀더로 SQL을 생성합니다.
결과에 "테이블명은 AI 추측 — 실제와 다를 수 있음" 경고가 표시됩니다.
SQL 문법 구조 파악이나 쿼리 초안 작성 참고용으로 활용할 수 있습니다.

```
출력 SQL (테이블 미등록):
SELECT <컬럼명> FROM <테이블명>
WHERE <조건컬럼> >= '값'
ORDER BY <날짜컬럼> DESC;
```

### 2-3. 테이블 & 컬럼 등록

실제 사용 중인 DB 테이블 구조를 미리 등록해두는 기능입니다.
등록 시 **테이블명**과 **컬럼 목록(컬럼명 + 설명)**을 입력합니다.

```
이름: 회원          테이블명: PLUSCMS.TA_MEMBER

컬럼명          설명
MEMBER_ID      회원 아이디
MEMBER_NM      회원 이름
PAY_AMT        결제 금액
JOIN_DT        가입 일자
```

- 브라우저에 저장되어 다음 접속 시에도 유지됩니다
- 여러 테이블을 등록해두고 필요한 것만 선택하여 사용합니다

#### "설명" 칸의 역할이 중요합니다

"설명"은 단순한 메모가 아닙니다.
AI가 한국어 질문과 실제 컬럼명을 연결할 때 이 설명을 참고합니다.

예를 들어 "회원 아이디가 123인 회원"이라고 입력하면,
AI가 설명에서 "회원 아이디"를 찾아 `MEMBER_ID` 컬럼으로 자동 매핑합니다.

> **설명은 실제 질문할 때 쓸 법한 한국어 표현으로 입력할수록 더 정확한 SQL이 생성됩니다.**

### 2-4. 선택된 테이블 컬럼 확인

테이블을 선택하면 상단에 칩(태그) 형태로 표시됩니다.
칩을 클릭하면 등록된 컬럼 목록이 펼쳐져 AI에게 전달되는 정보를 직접 확인할 수 있습니다.

```
[회원  PLUSCMS.TA_MEMBER  ▼]  ← 클릭

┌────────────────────────────────┐
│ 컬럼명         설명            │
│ MEMBER_ID     회원 아이디      │
│ MEMBER_NM     회원 이름        │
│ PAY_AMT       결제 금액        │
│ JOIN_DT       가입 일자        │
└────────────────────────────────┘
```

등록한 내용이 맞는지 확인하거나, 복잡한 JOIN 쿼리 전에 컬럼을 검토할 때 유용합니다.

### 2-5. 결과 신뢰도 뱃지

변환 결과 우측 상단에 테이블 등록 여부를 표시합니다.

| 상태 | 뱃지 |
|------|------|
| 등록된 테이블 선택 후 변환 | ✅ 등록된 스키마 기반 |
| 테이블 미등록/미선택 후 변환 | ⚠️ 테이블명은 AI 추측 — 실제와 다를 수 있음 |

### 2-6. SQL 복사 버튼

생성된 SQL을 클릭 한 번으로 클립보드에 복사합니다.
복사 후 DBeaver, SQL Developer 등 DB 툴에 바로 붙여넣을 수 있습니다.

## 4. 사용 흐름

### 처음 사용하는 경우 (온보딩)

```
1. 접속
    ↓
2. "테이블을 등록하면 실무에 맞는 SQL이 생성됩니다" 안내 확인
    ↓
3. [지금 등록하기] 클릭
    → 테이블 이름, 실제 테이블명(DB명) 입력
    → 컬럼명과 설명을 행 단위로 입력
    ↓
4. 저장하면 자동으로 선택됨
    ↓
5. 자연어 입력 → 변환하기 (또는 ⌘Enter)
    ↓
6. ✅ 실제 테이블명/컬럼명으로 SQL 생성
```

### 이미 테이블을 등록한 경우

```
1. 검색창에서 사용할 테이블 검색 후 선택
    ↓
2. 필요하면 칩 클릭 → 컬럼 목록 확인
    ↓
3. 자연어 입력 → 변환하기
    ↓
4. ✅ SQL 생성 → 복사 → DB 툴에 붙여넣기
```

### 테이블 없이 바로 사용하는 경우

```
1. 자연어 입력 → 변환하기
    ↓
2. ⚠️ <테이블명>, <컬럼명> 형태의 SQL 생성
    ↓
3. 직접 실제 테이블명/컬럼명으로 교체해서 사용
```

## 5. 기술 구성

### 전체 흐름

```
사용자 입력
    ↓
브라우저 (Next.js 앱)
    ↓  API 요청 (HTTPS)
서버 (Next.js API Route)
    ↓  API 요청 (Groq 서버로)
Groq AI 서버 (llama-3.3-70b 모델)
    ↓  JSON 응답
서버 → 브라우저 → 화면 표시
```

### 각 구성요소 역할

| 구성요소 | 쉬운 설명 |
|----------|-----------|
| **Next.js** | 웹 앱을 만드는 프레임워크. 화면(프론트엔드)과 서버(백엔드) 역할을 동시에 담당 |
| **React** | 화면의 버튼, 입력창 등 UI 요소를 만드는 라이브러리 |
| **Tailwind CSS** | 화면 디자인(색상, 여백, 크기 등)을 빠르게 적용하는 도구 |
| **Groq API** | AI 모델을 호출할 수 있는 서비스. 무료 tier 제공 |
| **llama-3.3-70b** | Groq에서 제공하는 AI 모델. 자연어 이해 및 SQL 생성 담당 |
| **Vercel** | 웹 앱을 인터넷에 배포하는 플랫폼. GitHub 연동으로 자동 배포 |

### AI에게 전달되는 정보 구조

테이블을 선택하면 아래 형태로 AI에게 전달됩니다.
"설명" 칸이 있어야 AI가 한국어 질문을 실제 컬럼명으로 정확하게 매핑할 수 있습니다.

```
TABLE: PLUSCMS.TA_MEMBER (회원)
COLUMNS:
- MEMBER_ID: 회원 아이디
- MEMBER_NM: 회원 이름
- PAY_AMT: 결제 금액
- JOIN_DT: 가입 일자
```

### API 키 보안 구조

```
❌ 잘못된 방식:
   브라우저 → Groq API (API 키가 소스코드에 노출)

✅ 현재 구조:
   브라우저 → 우리 서버(API Route) → Groq API
                    ↑
              API 키는 서버에만 보관
              외부에서 절대 보이지 않음
```

## 7. 설계 결정 이유

### Q. 왜 테이블 등록을 권장하나요? 없이도 쓸 수 있는데?

테이블 미등록 시 생성된 SQL은 ChatGPT에게 물어본 것과 차이가 없습니다.
이 도구의 핵심 가치는 "우리 회사 DB 구조를 아는 AI"에 있기 때문에,
테이블 등록을 자연스럽게 유도하는 것이 사용자에게도 더 유익합니다.

### Q. 테이블 미등록 시 왜 플레이스홀더를 쓰나요?

그럴듯한 가짜 테이블명(예: `USERS`, `ORDERS`)은 실제처럼 보여 혼란을 줄 수 있습니다.
`<테이블명>` 형태로 출력하면 사용자가 "내가 직접 채워야 한다"는 것을 즉시 인식합니다.

### Q. 컬럼 입력 시 "설명"은 왜 필요한가요?

DB 컬럼명은 보통 영문 약어입니다(예: `CLAMSTDUIQNO`).
사람이 자연어로 "청구고유번호가 ..."라고 입력했을 때 AI가 이 컬럼을 찾으려면 한글 설명이 필요합니다.
설명이 없으면 AI가 컬럼명만 보고 의미를 추측해야 하므로 오류 가능성이 높아집니다.

### Q. 왜 Vercel인가요? GitHub Pages는 안 되나요?

이 앱은 API 키를 서버에서 안전하게 관리해야 합니다.
GitHub Pages는 서버 기능이 없어 API 키가 소스코드에 노출되는 보안 문제가 생깁니다.
Vercel은 서버리스 함수를 지원하고 무료로 사용 가능해서 선택했습니다.

### Q. 왜 Groq인가요?

| 서비스 | 비용 | 속도 | 비고 |
|--------|------|------|------|
| OpenAI GPT-4 | 유료 | 보통 | API 키 필수 |
| Anthropic Claude | 유료 | 보통 | API 키 필수 |
| **Groq** | **무료 tier** | **매우 빠름** | llama 모델 제공 |

### Q. 테이블 정보는 어디에 저장되나요?

사용자 브라우저의 `localStorage`에 저장됩니다.
서버나 DB에 저장되지 않아 개인 정보 보호 측면에서 안전합니다.
단, 다른 기기나 다른 브라우저에서는 공유되지 않습니다.

## 8. 향후 개선 방향

### 단기 (바로 적용 가능)

- **DB 방언 선택**: MySQL / PostgreSQL / Oracle / MSSQL 문법 선택
- **예시 질문 버튼**: 초보 사용자를 위한 질문 예시 제공
- **히스토리**: 이전 변환 내역 저장 및 재사용

### 중기 (사내 도구로 확장 시)

- **SELECT 전용 제한**: DELETE, DROP 등 위험 쿼리 생성 차단
- **쿼리 공유 링크**: 생성된 쿼리를 URL로 공유
- **역방향 변환**: SQL → 한글 설명 (다른 사람이 짠 쿼리 해석)

### 장기 (팀 협업 도구로 확장 시)

- **팀 공유 테이블 스키마**: 팀원 간 테이블 정보 공유 (현재는 개인 브라우저 저장)
- **즐겨찾기**: 자주 쓰는 쿼리 저장 및 관리
- **사용 로그**: 누가 어떤 쿼리를 생성했는지 기록 (감사 목적)
